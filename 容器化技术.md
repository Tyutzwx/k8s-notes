## 一、核心概念与原理

### 1. 定义

容器化技术是基于**操作系统级虚拟化**技术，将应用程序及其依赖、配置打包成可移植的标准化单元（容器），实现跨环境快速部署与运行。

### 2. 底层核心技术（Linux 基石）

容器的隔离与限制能力，完全依赖 Linux 内核的两大核心机制:

| 技术机制                  | 核心作用                                                     | 常见应用场景                               |
| ------------------------- | ------------------------------------------------------------ | ------------------------------------------ |
| **Namespace（命名空间）** | 实现资源隔离，让容器拥有独立的进程、网络、挂载、用户等视图，互不干扰 | 隔离容器进程空间、独立网络栈、隔离文件系统 |
| **Cgroup（控制组）**      | 限制容器的资源使用上限，包括 CPU、内存、磁盘 I/O、网络带宽等，防止资源争抢 | 控制容器 CPU 使用率、内存上限、IO 优先级   |

### 3. 与虚拟机的核心差异（面试必背）

- **架构层面**：虚拟机通过 Hypervisor 模拟硬件，拥有独立完整操作系统；容器共享宿主机内核，仅封装应用与依赖。
- **资源开销**：虚拟机启动慢（分钟级）、占用高（GB 级）；容器启动快（毫秒 / 秒级）、占用低（MB 级）。
- **隔离性**：虚拟机强隔离（硬件级）；容器弱隔离（进程级），存在内核共享风险。

## 二、核心工具：Docker

### 1. 核心组件

- **Docker Engine**：核心运行时，包含守护进程 (dockerd) 和命令行工具 (docker)。
- **Docker Image（镜像）**：容器的只读模板，包含应用代码、依赖、环境配置，分层存储设计。
- **Docker Container（容器）**：镜像的运行实例，可启动、停止、删除，是隔离的运行环境。
- **Docker Registry（仓库）**：镜像存储与分发中心，公共仓库如 Docker Hub，私有仓库如 Harbor。

### 2. 核心操作流程

#### （1）镜像构建

- **Dockerfile**：定义镜像构建步骤的脚本，核心指令示例：

```
# 基础镜像
FROM centos:7
# 维护者信息
LABEL maintainer="yourname@example.com"
# 安装依赖
RUN yum install -y nginx
# 暴露端口
EXPOSE 80
# 启动命令
CMD ["nginx", "-g", "daemon off;"]
```

- 构建命令：`docker build -t 镜像名称:标签 Dockerfile路径`

#### （2）容器运行

- 基础运行命令：`docker run -d -p 宿主机端口:容器端口 --name 容器名称 镜像名称:标签`
- 关键参数解析：
  - `-d`：后台运行容器
  - `-p`：端口映射，实现外部访问容器内服务
  - `-v`：数据卷挂载，实现数据持久化与宿主机共享
  - `--restart=always`：容器退出时自动重启

#### （3）容器管理

- 查看容器：`docker ps [-a]`（-a 包含停止状态容器）
- 进入容器：`docker exec -it 容器ID/名称 /bin/bash`
- 停止 / 启动容器：`docker stop/start 容器ID/名称`
- 删除容器：`docker rm 容器ID/名称`（强制删除加 -f）

### 3. 镜像优化要点

1. **多阶段构建**：减小镜像体积，分离构建与运行环境

```
# 构建阶段
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o main main.go

# 运行阶段
FROM alpine:3.18
WORKDIR /app
COPY --from=builder /app/main .
EXPOSE 8080
CMD ["./main"]
```

1. **使用轻量基础镜像**：如 alpine、scratch，替代 ubuntu、centos
2. **清理冗余层**：合并 RUN 指令，减少镜像层数

## 三、容器编排：Kubernetes（K8s）

### 1. 核心定位

容器化应用的**集群管理平台**，实现容器的自动化部署、扩缩容、自愈、负载均衡与滚动更新。

### 2. 核心架构

#### （1）控制平面（Master 节点）

- **kube-apiserver**：集群统一入口，所有操作通过 API 进行
- **etcd**：集群状态存储，保存所有资源配置与数据
- **kube-scheduler**：容器调度，决定 Pod 运行在哪个节点
- **kube-controller-manager**：核心控制器，确保集群状态符合预期

#### （2）节点（Node 节点）

- **kubelet**：节点代理，负责 Pod 的创建、启动、监控
- **kube-proxy**：网络代理，实现 Service 的负载均衡与网络转发
- **容器运行时**：如 containerd、Docker（1.24 版本后默认移除 dockershim）

### 3. 核心资源对象

| 资源对象             | 核心作用                                            | 关键特性                                                     |
| -------------------- | --------------------------------------------------- | ------------------------------------------------------------ |
| **Pod**              | 最小部署单元，一个 Pod 包含一个或多个紧密关联的容器 | 共享网络、存储命名空间，是 K8s 调度的基本单位                |
| **Deployment**       | 无状态应用部署控制器                                | 支持滚动更新、回滚、扩缩容，确保应用副本数稳定               |
| **Service**          | 服务暴露与负载均衡                                  | 固定访问入口，实现 Pod 的负载转发，支持 ClusterIP、NodePort、LoadBalancer 等类型 |
| **Volume**           | 持久化存储                                          | 挂载到 Pod，实现数据持久化，支持本地存储、网络存储（NFS、Ceph）等 |
| **ConfigMap/Secret** | 配置管理                                            | ConfigMap 存非敏感配置，Secret 存敏感信息（如密码、密钥），避免硬编码 |

### 4. 核心工作流程

1. 编写资源配置文件（YAML 格式），定义 Deployment、Service 等资源
2. 通过 kubectl 提交配置：`kubectl apply -f 配置文件.yaml`
3. kube-apiserver 接收请求，存储到 etcd
4. 控制器监听资源变化，执行相应操作（如创建 Pod、调度节点）
5. kubelet 在节点上启动 Pod，确保应用正常运行

## 四、容器网络与存储

### 1. 容器网络模型

- CNI（容器网络接口）

  ：K8s 容器网络标准，主流插件：

  - **Flannel**：轻量简单，基于 VXLAN/Host-GW 实现网络互通
  - **Calico**：高性能，基于 BGP 路由，支持网络策略（NetworkPolicy）实现精细化访问控制

- **核心网络需求**：跨节点容器互通、Pod 独立 IP、集群内服务发现

### 2. 容器存储方案

- **本地存储**：emptyDir（临时存储）、hostPath（宿主机目录）
- 网络存储：
  - **Ceph**：分布式存储系统，支持块存储、文件存储、对象存储，适配 K8s 持久化需求
  - **NFS**：网络文件系统，简单易用，适合小规模共享存储
- **StorageClass**：K8s 存储类，动态供给持久化卷，无需手动创建 PV

## 五、容器和虚拟机

- **虚拟机（VM）**

  硬件级虚拟化，通过 Hypervisor 模拟 CPU、内存、网卡等硬件，每个 VM 运行**完整独立的操作系统与内核**

- **容器（Container）**

  OS 级虚拟化，基于 Linux Namespace 做隔离、Cgroup 做资源限制，**共享宿主机内核**，只打包应用 + 依赖库


------

### 二、核心区别（必背表格）

| 对比项          | 虚拟机                       | 容器                                       |
| :-------------- | :--------------------------- | :----------------------------------------- |
| 虚拟化层级      | 硬件虚拟化                   | 操作系统虚拟化                             |
| 内核            | 独立内核，完全隔离           | 共享宿主机内核                             |
| 启动速度        | 慢，分钟级                   | 快，秒 / 毫秒级                            |
| 资源占用        | 高，GB 级别                  | 低，MB 级别                                |
| 镜像体积        | 大（完整系统）               | 小（仅应用依赖）                           |
| 隔离性 / 安全性 | 强，接近物理机隔离           | 弱，进程级隔离，存在逃逸风险               |
| 跨 OS 运行      | 可 Windows/Linux 混跑        | 同内核才能运行（Linux 容器不能跑 Windows） |
| 适用场景        | 强隔离、传统应用、多 OS 环境 | 微服务、弹性扩缩容、CI/CD、云原生          |

------

### 虚拟机原理

- 依赖 **Hypervisor**（如 KVM、VMware、Xen）
- 虚拟出一套完整硬件，VM 内部独立引导、启动内核、运行系统。

### 容器原理（Linux 核心）

- Namespace

  ：实现隔离

  - PID：进程号隔离
  - NET：网络栈隔离
  - MNT：文件系统挂载隔离
  - UTS：主机名域名隔离
  - IPC：进程间通信隔离
  - USER：用户权限隔离

- Cgroup

  ：限制资源

  - 限制 CPU、内存、磁盘 IO、网络带宽等。

------


### 虚拟机优点

- 隔离强、安全性高
- 支持异构操作系统
- 成熟稳定，兼容性好

### 容器优点

- 轻量、启动快
- 环境一致性强（一次构建处处运行）
- 资源利用率高，适合高密度部署
- 便于 K8s 编排、弹性伸缩

------

## 五、经典面试题标准答案

### 1. 容器和虚拟机最大区别是什么？

**虚拟机是模拟硬件，有独立 OS 和内核；容器共享宿主机内核，只隔离应用进程。**

### 2. 为什么容器比虚拟机轻？

因为容器**不需要启动完整操作系统和内核**，省去大量资源开销，直接以进程形式运行。

### 3. 容器隔离性为什么不如虚拟机？

因为**共享同一个内核**，一旦内核出现漏洞，可能出现容器逃逸，影响宿主机和其他容器。

### 4. 两者是什么关系？

**互补关系，不是替代关系。**

- 公有云通常：物理机 → 虚拟机 → 容器
- 虚拟机提供强安全隔离，容器提供高效部署弹性。

### 5. 什么场景用 VM，什么场景用容器？

- 用 VM：强隔离、多系统、传统单体应用、安全性要求极高。
- 用容器：微服务、快速迭代、CI/CD、弹性扩缩容、云原生架构。
