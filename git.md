#git的操作

用**写周记 + 和同学合编班级手册**的生活化例子讲 Git 操作，全程大白话，不扯专业术语，看完就能懂核心逻辑：

假设你要做两件事：① 自己写周记，怕写坏了想撤回；② 和同桌一起编《班级手册》，俩人各写一部分，怕互相改乱，这就是 Git 要解决的问题，**本地仓库 = 你的笔记本**，**远程仓库 = 班级公用的网盘**，Git 就是帮你管笔记本 + 同步网盘的工具。

### 一、单人操作：自己写周记（对应单人开发 / 写运维脚本）

核心需求：随时保存进度、写砸了能回滚到之前的版本、想加新内容又不破坏原有内容。

#### 1. 准备笔记本（初始化本地仓库）

你拿了个新笔记本，写上 “我的周记本”，这对应：

`git init` → 在电脑文件夹里创建 Git 仓库，告诉 Git：“从现在开始，管这个文件夹里的所有文件”。

#### 2. 写第一篇周记并保存（新建文件 + 提交版本）

你在笔记本写了第 1 篇周记《周一上学》，写完怕丢，夹上书签标 “版本 1”，这对应两步（Git 的核心提交逻辑，必须走）：

① `git add 周记.txt` → 把写好的周记 “放进笔记本待保存区”，告诉 Git：“我要保存这个文件的修改”；

② `git commit -m "写了周一上学的周记"` → 正式保存，打版本标签，备注修改内容，Git 会生成唯一 “版本 ID”，相当于书签，以后想回滚直接找这个书签。

#### 3. 改周记 + 保存新版本（修改文件 + 再次提交）

你觉得第 1 篇写得不好，改了几句，写完再夹个书签标 “版本 2”，还是重复上面两步：

`git add 周记.txt` → `git commit -m "修改周一周记的细节"`，此时笔记本里有 ** 版本 1（原始）、版本 2（修改后）** 两个完整版本，Git 只存修改差异，不占额外空间。

#### 4. 写砸了，回滚到之前的版本（版本回溯）

你改到版本 3，发现越改越烂，想回到版本 1 的原始内容，直接翻到版本 1 的书签，把后面的内容划掉，对应：

① `git log` → 查看所有版本书签（版本 ID + 备注），找到版本 1 的 ID；

② `git reset --hard 版本1的ID` → 一键回滚，文件夹里的周记直接变回版本 1 的样子，简单粗暴。

#### 5. 想写新内容，又不想破坏现有周记（创建分支）

你想写《周末出游》的新周记，又怕写的过程中把原来的周一周记弄乱，于是在笔记本里折了一页，标上 “出游分支”，单独在这页写，原来的页数（主分支 main）不动，对应：

`git branch 出游分支` → 创建分支；

`git switch 出游分支` → 切换到新分支写内容，写完正常`add+commit`保存，此时主分支和出游分支互不影响。

#### 6. 新内容写好了，合并到原周记里（分支合并）

出游周记写完了，想把它放到主周记里，把折页的内容粘到原页数后面，对应：

`git switch main` → 切回主分支；

`git merge 出游分支` → 把出游分支的内容合并到主分支，此时主分支里既有周一周记，又有周末出游周记。

### 二、多人协作：和同桌合编《班级手册》（对应团队开发 / 维护运维配置）

核心需求：俩人各写一部分（你写班委名单，同桌写班级活动），同步到公用网盘，不互相覆盖，改冲突了能协商解决。

#### 1. 把公用网盘的手册拉到自己电脑（克隆远程仓库）

班级网盘里有空白的《班级手册》，你把它下载到自己电脑，同时把网盘的更新规则同步过来，对应：

`git clone 网盘地址` → 把远程仓库（网盘）的所有内容 + 版本记录，完整复制到本地，你和同桌都执行这一步，俩人本地都有一样的空白手册。

#### 2. 你写自己的部分，本地保存（单人开发提交）

你在本地手册里写 “班委名单”，写完正常`git add 班级手册.md` → `git commit -m "写完班委名单"`，保存到自己的本地仓库（笔记本）。

#### 3. 把你的内容同步到网盘（本地推送到远程）

你想让同桌看到你写的班委名单，把本地的修改上传到网盘，对应：

`git push` → 把本地的版本提交到远程仓库（网盘），此时网盘里的手册就有了班委名单。

#### 4. 拉取同桌写的内容（远程拉取到本地）

同桌写完了 “班级活动”，并推送到了网盘，你想把他的内容同步到自己电脑，对应：

`git pull` → 把网盘里的最新内容（同桌写的）下载到自己本地，此时你电脑里的手册既有班委名单，又有班级活动，自动合并。

#### 5. 俩人改了同一处，出现冲突（解决冲突）

最常见的情况：你和同桌同时改了 “班长名字”，你写的是 “张三”，同桌写的是 “张小三”，Git 无法判断谁的对，就会提示**冲突**，对应：

① `git pull` → 拉取时 Git 会提示 “冲突”，并在文件里标注冲突位置（用 <<<<<>>>>> 框起来，左边是你的内容，右边是同桌的）；

② 手动协商：你和同桌确认班长正确名字是 “张三”，手动把文件里的冲突标注删掉，只保留 “张三”；

③ 重新提交：`git add 班级手册.md` → `git commit -m "解决班长名字的冲突，确认是张三"` → `git push`，此时网盘里的内容就统一了。

#### 6. 保护网盘里的最终版本（主分支保护，大厂必做）

为了防止有人乱改网盘里的最终手册，班级规定：**不能直接改网盘里的内容，必须先在自己电脑写好，发给班长审核，审核通过后班长再合并到网盘**，这对应企业里的**MR/PR（合并请求）**：

你在本地写好内容后，不直接`git push`到主分支，而是推到自己的 “功能分支”，然后在 GitLab/GitHub 上提交 “合并请求”，让运维 / 开发负责人审核，审核通过后，负责人再把你的分支合并到主分支，避免主分支（线上可用版本）被改乱。

### 三、Git 核心操作总结（对应上面的例子，90% 场景够用）

| 通俗操作         | Git 命令                  | 适用场景                        |
| :--------------- | :------------------------ | :------------------------------ |
| 初始化笔记本     | `git init`                | 本地新建仓库                    |
| 下载网盘内容     | `git clone 仓库地址`      | 第一次获取远程仓库内容          |
| 标记要保存的文件 | `git add 文件名/`.`       | 准备保存修改（`.`是所有文件）   |
| 正式保存打标签   | `git commit -m "备注"`    | 本地记录版本                    |
| 本地传送到网盘   | `git push`                | 同步本地修改到远程              |
| 网盘同步到本地   | `git pull`                | 同步远程修改到本地              |
| 查看所有版本     | `git log`                 | 找历史版本、版本 ID             |
| 回滚到旧版本     | `git reset --hard 版本ID` | 改砸了恢复到之前的版本          |
| 创建新分支       | `git branch 分支名`       | 开发新功能 / 测试，不影响主分支 |
| 切换分支         | `git switch 分支名`       | 切换到不同分支工作              |
| 合并分支         | `git merge 分支名`        | 新功能完成，合并到主分支        |
| 查看当前状态     | `git status`              | 看哪些文件改了、是否有冲突      |

### 四、SRE 日常实操的对应场景（把例子换成工作）

把上面的 “周记 / 班级手册” 换成**Ansible Playbook、K8s YAML 配置、运维脚本、监控规则**，就是你日常的 Git 操作：

1. 自己写 K8s 部署脚本 → 单人操作，`init+add+commit`保存，创建分支测试新脚本；
2. 和团队一起维护 OpenStack 配置文件 → 多人协作，`clone`拉取远程配置，`push`上传自己的修改，`pull`同步同事的配置，解决冲突；
3. 脚本改坏了 → `git log`找历史版本，`reset`回滚到能用的版本；
4. 线上配置不能乱改 → 主分支保护，所有修改走分支 + 审核，再合并到主分支。

### 最后一句核心总结

Git 的本质就是：**给文件做 “可回溯的版本快照”，让单人开发能回滚、多人协作不冲突，本地操作自由，远程同步方便**，不用记太多复杂命令，掌握上面这 10 个，就能应对 SRE 日常 90% 的 Git 操作。



# Git 推送代码完整流程（本地 → 远程）

## 一、先检查本地是否配置 Git（第一次必做）
```bash
git config --global user.name "你的名字"
git config --global user.email "你的邮箱"
```

查看配置是否成功：
```bash
git config --list
```

## 二、生成并配置 SSH 密钥（必须做，否则无法推送）

1. 生成密钥
```bash
ssh-keygen -t ed25519 -C "你的邮箱"
```
连续按 3 次回车。

2. 查看公钥
Windows / Mac 通用：
```bash
cat ~/.ssh/id_ed25519.pub
```
把输出的一整串内容复制。

3. 把公钥粘贴到 GitHub/GitLab/Gitee
进入网站 → Settings → SSH keys → 粘贴 → 添加。
测试是否连通：
```bash
ssh -T git@github.com
```
出现 successfully 就是成功。

## 三、Git 推送完整流程（核心 6 步）
1. 进入项目文件夹
```bash
cd 你的项目路径
```

2. 初始化 Git 仓库（第一次才需要）
```bash
git init
```

3. 关联远程仓库（第一次才需要）
去 GitHub 新建仓库，复制 SSH 地址：
```bash
git remote add origin git@github.com:用户名/仓库名.git
```

查看是否关联成功：
```bash
git remote -v
```

4. 添加文件到暂存区
```bash
# 添加所有文件
git add .

# 或只添加某个文件
git add 文件名
```

5. 提交代码
```bash
git commit -m "这里写本次提交说明"
```

6. 推送到远程仓库
```bash
git push -u origin main
```

以后再次推送只需要：
```bash
git push
```

## 四、以后修改代码后的简化流程
你只需要记 3 行：
```bash
git add .
git commit -m "更新内容"
git push
```

## 五、最常见的 3 个问题
1. 提示 main 或 master 不存在
```bash
git branch -M main
```
再 push。

2. 推送失败：远程有更新
先拉取再推送：
```bash
git pull
git push
```

3. 想取消本次添加
```bash
git reset
```

## 总结
完整流程一句话：
配置 Git → 生成密钥 → 关联远程 → add → commit → push
